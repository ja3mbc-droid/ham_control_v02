# 006_2026-07-15_アーキテクチャ見直し案

## ham_control_v02 全体アーキテクチャ（素案）

### 目的

本書は ham_control_v02 の全体構成を整理するための設計素案である。

現在は FreeDV・WSJT-X・flrig・HAMLOG 等の連携機能を実装中であり、各モジュールの責務を明確にすることを目的とする。

---

# 基本方針

本プロジェクトは

「記憶より記録」

を基本理念とし、各機能を独立したモジュールとして構成する。

各モジュールは単一責務とし、将来 fldigi や他ソフトを追加しても既存コードへの影響を最小限に抑える。

# モジュール責務

## GUI

ユーザーインターフェース。

画面表示および操作のみ担当する。

---

## Controller

GUIからの要求を各機能へ振り分ける。

送信系の中心となる。

---

## Receiver

UDP・TCP・XML-RPC等の受信を担当する。

データ解析は行わない。

---

## Parser

受信データを解析し、構造体へ変換する。

例

- WSJT-X Protocol
- FreeDV UDP

---

## Adapter

各ソフト固有のデータ形式を共通形式へ変換する。

例

- QsoLogged
- FreeDV
- fldigi

↓

QsoRecord

---

## LogManager

共通ログを管理する。

出力先は

- ADIF
- CSV
- HAMLOG

などを担当する。

---

# 設計原則

- GUI は通信を行わない
- Controller は全体を制御する
- Receiver は受信のみ担当する
- Parser は解析のみ担当する
- Adapter はソフト依存を吸収する
- LogManager はログ管理のみ担当する

# 追記 (2026-07-15) Adapter実装方針の確定

Adapterは共通traitとして定義し、ソフトごとに個別実装する。

    trait LogAdapter {
        fn to_qso_record(&self) -> QsoRecord;
    }

    struct WsjtxAdapter { ... }
    struct FreeDvAdapter { ... }
    struct FldigiAdapter { ... }

fldigi等を追加する際、既存のAdapter実装には一切手を加えず、
新しいimplを追加するのみとする(単一責務原則の徹底)。

Adapterのインスタンスは LogManager のみが所有する。
Receiver / Parser 側では生成しない(二重生成の防止)。
