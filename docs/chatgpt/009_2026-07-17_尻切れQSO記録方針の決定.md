# 009: 尻切れQSO記録方針の決定 (2026-07-17)

## 経緯

JI4JGD局との交信終了後、矢継ぎ早に数局から呼ばれ、最後に呼ばれた局(JH7OTG)は
73/RR73が返らない「尻切れ」となった。この実データをきっかけに、以下の問題が
連鎖的に見つかり、修正された。

1. GUIの「WSJT-Xから読込」機能(`latest_qso()`方式)は最新1件しか見えず、
   パイルアップ中に完了した局のログが失われることが判明。
2. 対策として、ALL.TXT全体を再走査し完了済みQSOを即座にCSV/ADIFへ書き込む
   `catch_up_wsjtx()`とバックグラウンドポーラー(`wsjtx_poller.rs`)を実装。
3. 実機検証中、同一コールサインとの別日の交信が誤って1レコードに結合される
   バグを発見・修正(`extract_all_qsos()`にセッション分割ロジックを追加)。

## ユーザーの一言による方針転換

上記3の修正確認中、ユーザーから次のような趣旨のコメントがあった。

> 尻切れ(Incomplete)のQSOも、後でその局から交信の有無を問い合わせられた際の
> 裏付けとして残しておきたい。現在の「完了済みのみ記録する」ルールとは違うが、
> これは自分のこだわりだ。

この一言がきっかけとなり、`catch_up_wsjtx()`の設計を以下のように変更した。

- **Complete** (73/RR73確認済み): そのまま記録
- **Incomplete** (尻切れ・応答が途中で止まった): COMMENT1に`[73未確認]`を
  自動付記した上で記録する(削除・無視しない)
- **NoResponse** (RSTレポートすら交換されていない): 空振り受信ノイズとして
  記録対象外のまま

重複排除キーにも状態(Complete/Incomplete)を含めることで、「最初は尻切れとして
記録→後で73が来て完了」となった場合に、尻切れの記録を消さずに完了の記録を
別途追加できるようにした。

## 実機検証結果

2026-07-17実施分で、JH7OTG局(21.074MHz, FT8, 尻切れ)が

```
2026-07-17 04:34:00,2026-07-17 04:44:45,21.074,FT8,JH7OTG,-02,,[73未確認],
```

として正しく記録されることを確認。この一言がなければ、尻切れQSOは
「未確定だから記録しない」という設計のまま残り、後日の局照会に対応できる
記録が失われていた。
