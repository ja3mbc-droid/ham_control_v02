## 重要な発見(2026-07-09、要再設計)

hamlog.rs / wsjtx.rs が送信しているUDPメッセージは、**実際のHAMLOG/WSJT-Xの
仕組みと噛み合っていないことが判明**した。調査の結果:

- HAMLOGは port 2237 で自由形式のテキストを受信・解釈する機能を持たない。
  2237番は本来「WSJT-X自身が送信する側のポート」(マルチキャスト 224.0.0.1)。
- 実際のHAMLOG連携は、WSJT-X → JTAlert/JT_Linker(仲介ソフト) → HAMLOG という
  経路で、QSO成立時のデータのみが転送される。または HAMLOG が ALL.TXT を
  監視する(JT-Get's機能)方式。
- そもそも ham_control_v02 は FT8デコードやQSO成立判定を行っておらず、
  hamlog.rs が送っているのは「現在のリグ状態(周波数/モード/PTT)」であって
  「交信記録」ではない。HAMLOGが本来必要とする情報(コールサイン等)が
  含まれていない。

→ 現状の hamlog.rs / wsjtx.rs の "SENT OK" は「UDP送信自体は失敗しなかった」
という意味に過ぎず、実際にHAMLOG/WSJT-X側で処理されたことは確認できていない
(そもそも受け取る相手がいない可能性が高い)。
