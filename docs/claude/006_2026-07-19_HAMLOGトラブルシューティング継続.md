# 006: HAMLOG自動入力トラブルシューティング継続 — GNOMEセッションへの切替と新たな課題 (2026-07-19)

## 経緯

005で報告した「ウィンドウが画面に描画されない」問題の続き。前日、xdotool
`windowkill`がHAMLOG本体・`ham_control_v02`本体ごと巻き込んで強制終了し、
以後どのウィンドウも画面に表示されなくなっていた。

## 根本原因の特定

`~/.xsession-errors`を確認したところ、問題発生時刻(20:06:10頃)に
`xfce4-power-manager`・`xfce4-terminal`・`xfsettingsd`・`xfdesktop`などが
軒並み以下のエラーで同時にクラッシュしていた記録を発見。

```
libxfce4ui-WARNING: ICE I/O Error
libxfce4ui-WARNING: Disconnected from session manager.
Exiting due to channel error.
```

これは、Wine個別の不調ではなく、**XFCEデスクトップのセッションマネージャー
自体が`windowkill`の巻き添えでクラッシュした**ことを示している。
`ps aux | grep xfwm4`でウィンドウマネージャー(xfwm4)自体が動いていない
ことも確認し、これが「何度再起動しても直らない」描画問題の根本原因と特定した。

## セッション種別の切替による部分的な解決

ログアウト→ログインし直した結果、セッションがXFCEではなく**GNOME Shell
(Mutter)** に変わっていたことが`wmctrl -m`で判明した。GNOMEセッションでは、
HAMLOG本体・LOG-[A]・免許状Get's・JT-Get's等が正常に画面表示されるように
なり、前日からの描画トラブル自体は解消した。

## 新たに発覚した問題: 操作トリガーの非反応

GNOMEセッションでは、HAMLOG本体の以下の操作トリガーが反応しなくなった
(XFCEでは正常に動いていた)。

- 本体にマウスを乗せてEnterキーを押すことでLOG-[A]を開く操作
- 「免許状Get's」側でコールサインを入力し「To」ボタンでHAMLOG本体と連携する操作

フォーカスフォローズマウス設定の違いを疑い、以下を試したが改善しなかった。

```bash
gsettings set org.gnome.desktop.wm.preferences focus-mode 'mouse'
```

原因は未解決のまま。

## LOG-[A]の複数化・消失の繰り返し

GNOMEセッションでも、LOG-[A]を開く操作を複数回試みると、以前と同様に
ウィンドウが2個(1個は実体のない「幽霊ウィンドウ」)になる現象が再発した。
`xdotool windowactivate`で`XGetWindowProperty`エラーが出る方を幽霊ウィンドウ
と判断し、`windowclose`(`windowkill`より穏やかな閉じ方)で片付けようとしたが、
今回は**それでも全ウィンドウが消失する**巻き添えが発生した。

最終的に`wineserver -k`で全プロセスを終了し、HAMLOG本体を再起動したところで
本日の作業を終了した(本体が画面に表示されたかどうかは未確認のまま終了)。

## 現状

- `hamlog_auto.rs`のコード自体(xdotoolによるTab移動・入力ロジック)は
  実機ビルド済み・配線確認済みだが、**実際にLOG-[A]へ自動入力するテストは
  一度も実行できていない**。すべて環境面のトラブルシューティングで
  終わっている。
- ①GUI履歴一覧、②即記録、fldigi/WSJT-X/FreeDV連携など、既存の機能は
  すべて正常に動作している(今回の問題の影響を受けていない)。

## 次回への引き継ぎ

1. HAMLOG本体を起動し、まず**1回だけ**LOG-[A]を開く操作を試す
2. もし複数開いてしまった場合、個別のウィンドウ操作(`windowclose`含む)で
   減らそうとせず、`wineserver -k`で全体を再起動してやり直す方が結果的に
   早い可能性が高い(今回、`windowclose`ですら巻き添えが起きたため)
3. LOG-[A]が1個の状態を確認できたら、間を置かずすぐ`ham_control_v02`の
   自動入力テストに移る
4. 「マウスを乗せてEnter」操作がGNOMEで反応しない問題は未解決。もし
   このままLOG-[A]が安定して開けない場合、HAMLOGのメニューから直接開ける
   項目がないか確認する、またはXFCEセッションに戻すことも検討する
   (ただしXFCEに戻す場合はセッションマネージャーが健全か確認してから)

## 教訓

- `xdotool windowkill`だけでなく`windowclose`のようなウィンドウ操作
  コマンドも、Wineアプリに対しては予測しにくい巻き添えを起こすことがある
- デスクトップ環境の切替(XFCE⇔GNOME)は、描画問題を解決する一方で、
  アプリ側が前提とする操作トリガー(フォーカス挙動等)に別の非互換を
  生むことがある
