# 009_2026-07-17 fldigi対応とログソース選択機能

## 概要
006の残項目「⑦fldigi Adapter追加」に着手。fldigiのADIFパス誤りを修正し、
GUIに「WSJT-X / FreeDV / fldigi」を手動選択できるラジオボタンを追加した。

## 発見・修正した問題

### 1. fldigi ADIFパスの誤り
config.rsのデフォルトパスが `~/.fldigi/logbook.adif` だったが、
実際のfldigiは `~/.fldigi/logs/logbook.adif` (logsサブディレクトリ配下)に出力していた。
修正済み。

### 2. log_manager.rs: fldigi_logbook_pathがハードコードされていた
LogManager::new()内でFldigiLogAdapterに渡すパスが "~/.fldigi/logbook.adif" と
文字列決め打ちになっており、config.rsの値が一切使われていなかった。
RustはシェルのようにHOME展開する`~`を解釈しないため、このままでは
常にファイルが見つからず失敗する状態だった。
LogManager::new()にfldigi_logbook_path引数を追加し、main.rsから
cfg.fldigi_logbook_pathを渡すよう修正して解消。

## 設計変更: ログソースの自動判定 → 手動選択

これまでlatest_qso()は FreeDV → WSJT-X → fldigi の優先順位で
自動的に最初に見つかったデータを返す仕組みだった。しかし実運用では
「どのソフトのログを読みたいか」をユーザーが選びたい、という理由から、
GUIに「ログソース: ○WSJT-X ○FreeDV ○fldigi」のラジオボタンを追加した。

- log_manager.rsに `latest_qso_by_source(&self, source: &str)` を新設
- ui.rsのApp構造体に `log_source_selected: String` フィールドを追加
- ボタンは「ALL.TXTから読込」→「選択ソースから読込」に改名
  (3ソース全部を読む仕様になったため、WSJT-X専用の旧名が実態と合わなくなっていた)
- log_statusのメッセージも選択中のソース名を動的に表示するよう変更

## 実機確認結果

- ラジオボタンの表示・切り替え・ボタン名変更、いずれも正常動作を確認
- fldigi: 正しいパスを読みに行っている
  (`[Fldigi] reading /home/ja3mbc/.fldigi/logs/logbook.adif` ログ確認)
  が、fldigi側にヘッダーのみでQSOレコードが1件も無いため、
  「該当データが見つかりません」という正常な空振り動作を確認
- FreeDV: プロセス再起動によりメモリ上の最新QSO(Mutex保持)がリセットされており、
  再起動後に新規QSOを確定していないため空振りを確認(正常動作)
- WSJT-X: wsjtx_log.rsは自局コールサイン(JA3MBC)が関わる行のみを対象とする設計のため、
  直近で自局が実際に交信していない状態では空振りを確認(正常動作、バグではない)

いずれも「データが存在しない場合に何も表示しない」という設計通りの
正常な空振り動作であり、ロジック自体は正しいと判断。
実際の3ソースそれぞれでのデータ入り確認は、次回それぞれで交信した際に行う。
