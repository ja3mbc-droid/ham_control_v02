007/008でxdotool自動入力を断念した結論を受け、今日は残りの2案を実装した。

## ②案: WSJT-Xの手動入力まで完成(commit a5d849e)

一覧のボタンを、xdotool自動入力呼び出しから以下の2つに分離した。

- 「入力欄へ読込む」: `apply_qso_record()`を呼び、CALL/RST等をGUIの入力欄に
  表示するだけ(HAMLOGへは触れない)
- 「済にする」: `sent_to_hamlog`へキーを追加し、一覧から非表示にする

`sent_to_hamlog`は`~/ham_control_v02_sent_to_hamlog.txt`(1行1キー)に永続化する
ようにし、アプリ再起動をまたいでも消えないようにした(`config.rs`に
`sent_to_hamlog_path`を追加、`HAM_SENT_TO_HAMLOG_PATH`で上書き可)。

実機確認: 「入力欄へ読込む」→CALL欄等の反映、「済にする」→一覧から消える、
アプリ再起動後も戻ってこない、の3点をすべて確認済み。

## 副産物: fldigiのCW自動デコードに関する発見

fldigiの一覧に、同一交信と思われる行が2件表示される事象があった。
`~/.fldigi/logs/logbook.adif`を直接確認したところ、アプリ側のパース
バグではなく、**fldigi自身が実際にそう記録していた**ことが判明。

```
<CALL:10>J H1 N K A
```

CWの自動デコードが文字間の間隔を単語区切りと誤認識し、"JH1NKA"を
1文字ずつ分解してスペースを挟んで記録してしまったと見られる。また
TIME_OFFが数秒違う別レコードとして2件残っていたため、記録自体が
2重になっていた可能性もある(fldigi側の「Log QSO」を誤って2回押した、
または自動記録が2回走った、など)。

対応方針: 今回は「何もしない」を選択。実データがそうなっている以上、
一覧に2件出るのは正しい表示であり、手入力の際に運用者が気づいて
片方だけ「済にする」で処理すればカバーできる、という判断。CALLの
空白除去(`J H1 N K A` → `JH1NKA`)をパース時に追加する案も検討したが、
今回は見送った。
