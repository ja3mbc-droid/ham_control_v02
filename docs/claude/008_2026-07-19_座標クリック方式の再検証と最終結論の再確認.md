# 008: 座標クリック方式によるLOG-[A]二重窓問題の再検証 (2026-07-19)

## 経緯

007で「xdotoolによるGUI自動化アプローチは断念する」と結論したが、
その後の別セッション(Claude Chat)で「ウィンドウ位置を固定してしまえば
二重窓が開いても実害なく回避できるのではないか」というアイデアが浮上し、
座標ベースのクリック方式について再検証を行った。

## 前提確認

- `$XDG_SESSION_TYPE` → `x11`。GNOME ShellだがWaylandではなくXorgセッション
  であることを確認。xdotoolの合成入力が制限を受ける懸念はないと判明。
- 実際にデュアルモニタ構成であり、HAMLOG本体・LOG-[A]画面がどちらの
  モニタに開くかは毎回まちまちであることを確認。

## 検証した仮説と結果

| 仮説 | 検証方法 | 結果 |
|---|---|---|
| windowmoveで位置固定すれば二重窓問題を回避できる | `xdotool windowmove <id> 100 100`を実行後、`getwindowgeometry`で確認 | 指定座標に移動せず、元の位置のまま。**windowmove自体がこの環境で機能していない** |
| 全角/半角IME変換候補ウィンドウが2個目の正体では | `getwindowname`で両IDのタイトルを比較 | 両方とも正規の「ＬＯＧ-[Ａ]」というタイトルだった。**仮説は誤り** |
| wmctrl登録の有無で本物/幽霊を判別できるのでは | `wmctrl -l`と`xdotool search`のID突き合わせ | wmctrl登録される方(小さいウィンドウ, 723x210)と、実際にクリックが成功した方(大きいウィンドウ, 751x276)が食い違う結果となり、**判別基準として使えないと判明** |
| 今日蓄積したゾンビ窓では | `wineserver -k`でプロセスを完全終了し、クリーンな状態から再起動してLOG-[A]を1回開く | 新しいID2個(過去と異なる値)が再度出現。**「LOG-[A]を開くたびに毎回2個生成される」構造的な挙動と判明**(ゾンビ窓説は誤り) |
| ウィンドウ左上からの相対オフセットでCall欄を計算すれば座標ベースで安定入力できるのでは | 1回目のテストで算出した相対オフセット(+64, +79)を、別の機会に開いた新しいウィンドウ原点に適用してクリック | 空振り。**同じ相対オフセットの計算式が毎回通用するわけではなく、開くたびに内部レイアウト状態が微妙に変動する** |

## 最終結論(再確認)

座標クリック方式は、二重窓問題そのものを「実害なく回避する」ことを
期待して検証したが、以下の複合的な不安定要因により、この環境では
安定した自動化の実現は困難と判断した。

1. `windowmove`が機能しない(位置を固定できない)
2. LOG-[A]を開くたびに、毎回再現性を持って2つのX11ウィンドウが生成される
   (原因未特定。Wine内部の描画/管理の仕組みによるものと推測)
3. どちらが「実際に入力を受け付ける本物」かは、wmctrl登録の有無・
   ウィンドウサイズ・クリック成否のいずれの手がかりでも一貫した判別が
   できなかった
4. ウィンドウ内のフィールド位置(相対オフセット)も、開くたびに微妙に
   変動し、決め打ちの座標計算が通用しなかった

007の結論(xdotool自動化アプローチの断念)を覆す材料は得られず、
むしろ「なぜ困難なのか」をより具体的に説明できる形で結論が補強された。

## 今後の方針(変更なし)

007と同じく、HAMLOGとの連携は以下の機能で運用を続ける。

- ①GUI履歴一覧(`recent_wsjtx_qsos()`)
- ②即記録(`catch_up_wsjtx()`)
- ⑥手入力支援(CSV/ADIF/GUI表示を見ながら運用者が手入力)

次の改善候補として、一覧のボタンの役割を「xdotool自動入力を試みる」
から「手入力完了をチェック済みにする(`sent_to_hamlog`へ追加するだけ)」
に変更する案が挙がっている。これは次回実装予定。
