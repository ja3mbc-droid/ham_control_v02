# 005: HAMLOG自動入力(xdotool)の実装と実機トラブルシューティング (2026-07-18)

## 経緯

前日(2026-07-17)の①GUI履歴一覧実装に続き、ユーザーから
「一覧クリックでHAMLOG本体の入力画面に直接入力したい」との要望があった。
HAMLOG(Wine上でTurbo HAMLOG/Win Ver5.47cとして動作)は外部連携APIを
持たないため、xdotoolによるウィンドウ操作の自動化(人間がキーボードで
打つ動作を代行する方式)で実現することにした。

## 実機調査で判明した仕様

- 新規QSO入力画面はウィンドウタイトル` ＬＯＧ-[Ａ]　　　`(全角文字・全角
  スペースを含む)
- Tab移動順: Call → Date → Time → His(RST RCVD) → My(RST SENT) → Freq
  → Mode → Code → G.L → QSL → His Name → QTH → Remarks1 → Remarks2 → Save
- Call欄にコールサインを入力してEnterを押すと、HAMLOGが局データベースを
  検索し、過去に交信があれば交信履歴、なければユーザー登録内容を自動展開する
- Save欄にフォーカスがある状態でEnterを押すと保存される

## 実装方針

「下書きを自動で埋めて、最終確認・修正・保存は人間」という役割分担とし、
xdotoolによる自動入力は**Save欄の手前で停止し、Enterは押さない**設計とした。
これは、送信前に運用者がRST等を微調整したり、Remarks欄に加筆したりする
実際の運用フローを尊重するための、ユーザーとの合意事項。

### 追加・変更したファイル

- `src/hamlog_auto.rs`(新規): xdotoolを子プロセス呼び出しし、上記の
  自動入力フローを実装。ウィンドウが0個/複数個見つかった場合はエラーを
  返し、誤って別ウィンドウに送信する事故を防ぐ。
- `src/ui.rs`: 「直近の交信一覧」の行クリック時、`hamlog_auto::send_qso_to_hamlog()`
  を呼ぶよう変更。送信に成功したQSOは`sent_to_hamlog`(HashSet、アプリ
  再起動でリセットされる非永続)に記録し、一覧から除外する。
- `src/main.rs`: `hamlog_auto`モジュールを登録。

実機ビルド確認済み(warningのみ、エラーなし)。関数の配線も`grep`で確認済み。

## 実機テスト中に発生したトラブル

1. LOG-[A]画面が2個開いた状態でテストしたところ、安全策(複数見つかった
   場合は送信しない)が正しく発動し、エラーメッセージが表示された。ここまでは
   想定通りの動作。
2. 片方のLOG-[A]画面を閉じようとして`xdotool windowkill`を使用したところ、
   意図に反してHAMLOG本体・`ham_control_v02`本体ごと強制終了してしまった。
3. その後、何度再起動してもウィンドウが画面に描画されない状態が継続。
   `wmctrl -l`にはウィンドウが存在する(desktop番号が`-1`)が、
   `windowmap`/`wmctrl -a`/ワークスペース切り替えを試しても表示されず。
4. OS再起動(`sudo reboot`)を1回実施したが、再現(表示されないまま)。
5. セッション上限に達したため、原因未特定のまま作業を中断。

## 次回への引き継ぎ

1. もう一度OS再起動を試す
2. HAMLOGを1回だけ起動し、**まず画面に表示されることを確認してから**
   次のステップに進む(表示されない場合はここで原因調査を優先する)
3. 表示されたらLOG-[A]画面を1つだけ開く
4. `ham_control_v02`を起動し、一覧クリックでのHAMLOG自動入力を実際にテストする
5. 動作確認できたら、`hamlog_auto.rs`関連の変更をcommit・push

`windowkill`のような強制終了コマンドは、意図しない範囲まで巻き込む
リスクがあるため、次回以降はウィンドウを閉じる際は極力手動(×ボタン)
または`windowclose`を優先する。

## 教訓

xdotoolによるGUI自動化は強力だが、ウィンドウ管理系のコマンド
(`windowkill`等)は影響範囲が予想しにくく、慎重な検証環境
(閉じてよいウィンドウで先に試す等)が必要だと分かった。
